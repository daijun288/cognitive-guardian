# Cognitive Guardian AI Agent Instructions

作为接入了 Cognitive Guardian MCP 的 AI 编程助手，请遵循以下核心准则来使用本系统。Cognitive Guardian 的核心价值是**“为你补充跨层级、跨文件的视野盲区”**，而不是替代你的语义理解和决策能力。

## 🎯 核心认知

- **你看到的只是冰山一角**：即使你理解了当前文件，也无法轻易推断出 MyBatis XML 映射或 Vue 模板中对当前 Java 后端接口的隐式调用。
- **Guardian 是静态雷达**：它基于 AST 和静态索引构建依赖图，能补充你缺失的全局拓扑关联，但**看不见动态代理、反射和动态 SQL**。
- **最终判断在你**：Guardian 输出的是“可能受影响的风险清单”以及相关度评分（complexity/impact）。你需要结合上下文语义，排除泛化误报，判断哪些是真正的风险。

## 🚥 何时触发工具 (Trigger Conditions)

### ✅ **必须使用的场景 (MUST USE)**

1. **重构与签名修改**：当你准备重命名类/方法、修改方法参数签名、更改业务核心接口时。
   👉 必须先调用 `check_relevance` 扫描影响面。
2. **跨栈/深层全景修改**：修改了涉及前后端交互的 Controller，或者涉及 MyBatis 的 Mapper 接口时。
3. **批量与高风险提交流程化**：当你即将一次性修改多个文件或处理高耦合区域代码时。
   👉 应当在实际修改方案敲定前，调用 `assess_bulk_impact` 评估整体风险。

### ⚠️ **需要手动同步的场景 (MUST SYNC)**

- **索引时效更正**：当你刚刚大量修改了代码，紧接着又要基于新代码结构做依赖分析时。
  👉 Guardian 的后台防抖可能有延迟，必须先强制调用 `sync_file` 更新指定文件的 AST 索引。

### ❌ **禁止使用的场景 (DO NOT USE)**

*（避免过度工程化，降低用户等待时间）*
1. **纯新增逻辑**：从头编写新类、新方法、新独立功能，且无历史依赖时。
2. **微小/封闭域修改**：修改一段日志输出、调整一个局部的 if/else 逻辑、或者修改孤立的配置项。
3. **单栈浅层修改**：如果是纯前端或纯后端不涉及对外暴露接口的简单功能迭代。
4. **你已拥有全局上下文**：如果对话中你和用户已经完整梳理并提供了所有相关的调用链文件。

## 🛠 最佳实践工作流

当用户提出一个复杂修改请求（如：“重构用户登录逻辑”）：
1. **预查潜在盲区**：对核心入口使用 `check_relevance` 扫描。重点关注输出中的 `isMyBatisSql` 或 `isCrossStack` 标记。
2. **结合语义判断**：对 Guardian 给出的影响列表，用你的代码语义理解能力过滤误报。
3. **执行修改**：根据过滤后的真实依赖关联编写代码。如果修改了接口，记得同步更新受影响的 XML 或前端组件。
4. **(如果刚改完又需要查)**：调用 `sync_file` 确立最新认知基线。
